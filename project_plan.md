# プロジェクト計画

## 1. 目的

本プロジェクトは、テニスの試合動画を大規模に解析し、統計情報を取得して分析を円滑にするインフラを提供することを目的とします。将来的には、AI自体が分析を行う可能性も視野に入れていますが、現段階では主に分析作業を支援するシステムを目指します。

## 2. 手段

-   **データソース**: インターネット上で公開されている試合動画を主な解析対象とします。このため、単一カメラからの映像解析を前提とします。
-   **コア技術**: 動画から以下の要素を検出します。
    -   ボール (ball)
    -   コート (court)
    -   プレイヤー (player)
    -   姿勢 (pose)
-   **データ処理**: 検出された各要素を時系列データとして扱い、特定のイベントを検出します。
-   **システム構成**: バックグラウンドシステムで動画解析とイベント検出を行い、その結果をユーザーに提供する形態を想定しています。
-   **現状**: ボール、コート、プレイヤー、姿勢の検出モジュールは基本的な実装が完了していますが、拡張性と保守性を高めるためのリファクタリングの余地があります。

## 3. 技術スタックと開発方針

### 3.1. 機械学習フレームワーク

-   **Pytorch-Lightning**: 機械学習モデルの学習には Pytorch-Lightning を全面的に採用します。これにより、学習コードの定型化、再現性の確保、ハードウェアアクセラレーションの容易な利用を目指します。

### 3.2. タスクレベルのモジュール構成

各解析タスク（例: `src/ball`, `src/court` など）は、以下の標準化されたモジュール構成に従うことを目指します。これにより、タスク間の整合性を保ち、開発効率を向上させます。

-   `dataset/`: `torch.utils.data.Dataset` を提供し、データセットの読み込みと前処理を担当します。
-   `models/`: `torch.nn.Module` を提供し、ニューラルネットワークモデルの定義を担当します。
-   `lit_module/`: Pytorch-Lightning の `LightningModule` および `LightningDataModule` を提供し、学習ループ、検証ループ、テストループ、および関連するデータローダーの管理を担当します。
-   `main.py` (または類似のスクリプト): 上記モジュールを統合し、学習処理全体を統括します。
-   `predictor.py`: 各タスクに共通のインターフェースを持つ推論サブモジュールを提供します。これにより、推論処理の標準化を図ります。

### 3.3. 推論処理の統合

-   `scripts/infer.py`: 各タスクの `predictor.py` を利用し、動画全体の解析処理を統合的に実行するスクリプトを配置します。

### 3.4. ファイル管理と再現性

学習プロセスにおけるログ、チェックポイント、設定ファイルの管理は、開発の効率性と再現性に直結するため、以下の指針で統一感を持たせます。

-   **ログ**: 学習の進捗、メトリクス、エラー情報などを記録します。
-   **チェックポイント**: モデルの重みや学習状態を定期的に保存し、学習の再開や特定時点でのモデル評価を可能にします。
-   **設定ファイル**: ハイパーパラメータ、データパス、学習オプションなどを外部ファイル (`configs/` ディレクトリ以下に格納) で管理し、ハードコーディングを避けます。
    -   **課題**: 現在の `configs/` の構成は改善の余地があり、より体系的で理解しやすい構造を目指します。
-   **基本ルール**: ファイル管理に関する詳細なルールは、`.cursor/rules/pl_training.mdc` (または今後整備するドキュメント) を参照し、これに従います。不足している点や改善点があれば、随時この計画文書や関連ドキュメントに追記・修正します。
